Необходимо закешировать следующие части системы:
    1. MES для удобства работы операторов (ускорения их работы, снижения общей нагрузки на систему)
    2. CRM для кеширования пользовательских данных и использования их без похода в БД (Shop DB)
    3. БД с  3д изделиями (необходимо посчитать что выйдет дешевле, походы в БД и сетевые расходы или хранение изображений, которые в работе в кеше)

Основной мотивацией для внедрения кеширования является
    1. ускорение работы системы - повышение удобства работников и клиентов
    2. снижение нагрузки на БД, а значит уменьшение ситуаций, когда может произойти отказ системы при повышении обращений пользователей
    3. проще масштабировать системы, а значит увеличение единовременного количества пользователей, для которых не будет появляться ошибки (при стабильной работе системы)
    4. уменьшение затрат на ресурсах БД

Предлагаемое решение
    1. Использование Redis для серверного кеширования.
    2. Паттерн Write-Through для гарантии актуальности данных. В нашем случае требуется высокая согласованность данных и максимальное исключение при работе с кэшем. Небольшие задержки при ожидании ответа от БД будут менее критичными, нежели неконсистентные данные. Достаточное количество операций записи при обновлении статусов, изменении товара, изменнии данных клиента в самом товаре
    3. Программная инвалидация

Стратегия инвалидации
    Программная инвалидация. Необходима гибкая система управления кешем, в соответствии с логикой приложения. Используем данную стратегию по причинам сложных бизнес-процессов связанных с взятием в работу и передачу по этапам готового изделия, когда для каждого мастера требуется согласованные данные и каждое действие системы по разному влияет на кэшироавние и его изменения
